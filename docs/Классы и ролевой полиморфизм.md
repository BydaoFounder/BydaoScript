# **Ролевой полиморфизм (Roles) в BydaoScript**
## **1. Общая концепция**
В BydaoScript нет отдельной сущности "интерфейс". Есть только классы. Класс может содержать **пустые методы** — методы без реализации. Такой класс задаёт контракт, который должны выполнить другие классы.

Класс может находиться в двух типах отношений с другими классами:

- **Наследование** (:) — класс расширяет другой класс. Наследование всегда **одиночное**.
- **Роль** (as) — класс берёт на себя обязательство выполнить контракт, заданный другим классом. Ролей может быть **множество**.
-----
## **2. Пустые методы (контракты)**
Пустой метод объявляется без тела:

class Logger {

`    `pub func log(message: String);

`    `pub func setLevel(level: Int);

}

Такой класс нельзя инстанциировать — он содержит незавершённые объявления. Это не "интерфейс" в смысле отдельной сущности, а обычный класс, который просто не дописан. Любой класс, имеющий пустые методы (свои или унаследованные), считается абстрактным и не может быть создан через new.

-----
## **3. Наследование (:)**
Наследование передаёт всё — и реализацию, и пустые методы.

class FileLogger : Logger {

`    `// log() и setLevel() всё ещё пустые — FileLogger тоже абстрактный

}

class ConsoleLogger : Logger {

`    `pub func log(message: String) {

`        `print("[LOG] " + message);

`    `}

`    `pub func setLevel(level: Int) {

`        `// реализация

`    `}

}

ConsoleLogger реализовал все пустые методы — теперь он конкретный, можно создавать объекты.

Правила:

- Наследование всегда одно (один родитель)
- Наследуются все поля и методы (включая пустые)
- Можно переопределять любые методы (и пустые, и реализованные)
-----
## **4. Роли (as)**
Класс может заявить, что он выполняет роль (контракт) другого класса, даже если он не является его наследником. Это делается через ключевое слово as.

class Hydroplane : Airplane as Boat {

`    `// ...

}

Запись означает:

- Класс Hydroplane наследует Airplane (один родитель)
- Класс Hydroplane **обязуется выполнить контракт** класса Boat

Ролей может быть несколько:

class Amphicar : Vehicle as Boat as Car {

`    `// ...

}
### **4.1 Реализация роли**
Для каждой роли, указанной через as, в классе должен быть определён **блок реализации**:

class Hydroplane : Airplane as Boat {

`    `// основная часть класса

`    `pub func takeoff() { ... }



`    `// реализация роли Boat

`    `Boat {

`        `pub func float() {

`            `// как этот объект плавает

`        `}



`        `pub func enginePower(): Real {

`            `// используем метод самолёта

`            `return owner.enginePower();

`        `}

`    `}

}

Блок Boat { ... } содержит реализацию **всех пустых методов**, объявленных в классе Boat. Если в классе Boat есть методы с реализацией, их можно не переопределять — они наследуются в роль автоматически. Но при желании их тоже можно переопределить внутри блока.

Ключевое слово owner внутри блока роли ссылается на **весь объект целиком** (включая его основную часть и все остальные роли). Это позволяет обращаться к методам основного класса или другим ролям.

-----
## **5. Представления (Views)**
Объект, реализующий несколько ролей, имеет множество **представлений**. По умолчанию переменная содержит представление **основного класса** (того, который после :).

let h = Hydroplane();      // h "видит" только методы Airplane и Hydroplane

h.takeoff();               // OK

h.float();                 // ОШИБКА — в этом представлении нет метода float

Чтобы получить представление объекта в определённой роли, используется синтаксис obj.Role():

let b = h.Boat();          // b "видит" только методы, определённые в роли Boat

b.float();                 // OK

b.takeoff();               // ОШИБКА — в представлении Boat нет метода takeoff

Представления:

- **Не создают новый объект** — это та же сущность, но с другим "входом"
- Передаются по ссылке (как и все объекты в языке)
- Не имеют собственного состояния — состояние принадлежит объекту
- Позволяют явно указать, в какой роли объект используется

Можно получать представления разных ролей от одного объекта:

let boatView = h.Boat();

let carView = h.Car();

let alsoBoat = carView.Boat();  // от представления Car можно получить Boat — объект тот же

-----
## **6. Правила видимости и вызовы**
### **Из основного класса в роль**
Из основного класса методы роли **не видны напрямую**. Если нужно вызвать метод роли, следует получить соответствующее представление или использовать квалифицированный синтаксис (если будет введён).
### **Из роли в основную часть**
Внутри блока роли через owner доступны:

- все методы основного класса
- методы других ролей (через owner.OtherRole.method() или через получение представления)
### **Из роли в другую роль**
Можно получить представление другой роли через owner:

Boat {

`    `pub func move() {

`        `owner.Car().move();  // делегирование другой роли

`    `}

}

-----
## **7. Конфликты и неоднозначности**
### **Конфликт имени метода в основной части и в роли**
Это разные пространства имён. Метод move() в основном классе и метод move() в роли Boat — это разные методы, принадлежащие разным представлениям. Конфликта нет, потому что вызов всегда идёт через конкретное представление.
### **Конфликт между двумя ролями**
Если две роли требуют метод с одним именем, программист **обязан** реализовать его в обоих блоках. Каждая реализация может быть своей:

Boat {

`    `pub func move() { ... }  // как плаваем

}

Car {

`    `pub func move() { ... }  // как едем

}

Для избежания дублирования можно делегировать:

Car {

`    `pub func move() = Boat.move();  // используем ту же реализацию

}
### **Циклические вызовы**
Программист сам отвечает за то, чтобы не создавать бесконечную рекурсию. Компилятор не защищает от:

Boat {

`    `pub func move() {

`        `owner.Car().move();  // если Car.move() вызовет Boat.move() — будет цикл

`    `}

}

Это считается ошибкой программиста, как и бесконечный цикл while (true);

-----
## **8. Типизация и приведение**
### **Проверка типов**
Класс, реализующий роль, **не является** наследником класса-роли. Hydroplane не находится в отношении подтипа к Boat. Это разные типы.

func processBoat(b: Boat) { ... }

let h = Hydroplane();

processBoat(h);              // ОШИБКА — неявное преобразование запрещено

processBoat(h.Boat());       // ОК — явное получение представления
### **Явное приведение**
Если необходимо проверить, можно ли объект использовать в данной роли, используется синтаксис as (в выражениях):

let b = h as Boat;  // если h реализует Boat, то b — представление Boat, иначе ошибка

Или безопасный вариант с проверкой:

if h is Boat {

`    `let b = h as Boat;  // теперь точно можно

}

-----
## **9. Конструкторы и инициализация**
Если класс-роль имеет конструктор, он **не вызывается автоматически**. Роль — это только контракт на методы, а не часть иерархии объектов. Инициализация состояния, необходимого для методов роли, лежит на программисте.

Возможный вариант (если потребуется):

Boat {

`    `// псевдо-конструктор роли — вызывается вручную из конструктора основного класса

`    `pub func initBoat() {

`        `// инициализация полей, нужных для работы роли

`    `}

}

Поля для хранения состояния роли объявляются в основной части класса и используются в блоке роли.

-----
## **10. Наследование классов с ролями**
При наследовании всё просто:

class FlyingBoat : Hydroplane {

`    `// наследует реализацию роли Boat от Hydroplane

`    `// можно переопределить блок Boat, если нужно

`    `Boat {

`        `// новая реализация для этой роли

`    `}

}

Если класс-потомок хочет изменить реализацию роли, он может переопределить соответствующий блок. Если не переопределяет — используется реализация предка.

-----
## **11. Копирование объектов**
Все объекты передаются по ссылке. Для создания копии используется явный метод clone():

let h1 = Hydroplane();

let h2 = h1.clone();      // полная копия, включая все роли

let b1 = h1.Boat();       // представление Boat от оригинала

let b2 = h2.Boat();       // представление Boat от копии (независимое)

clone() создаёт глубокую копию объекта со всем его состоянием. Представления клонированного объекта ведут на новый объект.

## **12. Пример полного использования**

// Контракты

class Logger {

`    `pub func log(msg: String);

}

class Serializable {

`    `pub func toJSON(): String;

`    `pub func fromJSON(data: String);

}

// Базовый класс

class Service {

`    `pub func start() { print("Service started"); }

}

// Конкретный класс с ролями

class UserService : Service as Logger as Serializable {



`    `// данные

`    `private users: List<String>;



`    `// конструктор

`    `pub init() {

`        `users = [];

`    `}



`    `// методы основного класса

`    `pub func addUser(name: String) {

`        `users.append(name);

`        `// используем свою роль логирования

`        `owner.Logger().log("User added: " + name);

`    `}



`    `// реализация роли Logger

`    `Logger {

`        `pub func log(msg: String) {

`            `print("LOG [" + Time.now() + "]: " + msg);

`        `}

`    `}



`    `// реализация роли Serializable

`    `Serializable {

`        `pub func toJSON(): String {

`            `return users.toJSON();

`        `}



`        `pub func fromJSON(data: String) {

`            `users = List.fromJSON(data);

`            `owner.Logger().log("Users loaded from JSON");

`        `}

`    `}

}

// Использование

let service = UserService();

service.start();

service.addUser("Alice");

// Работа с ролью Logger

let logger = service.Logger();

logger.log("Hello from logger");

// Работа с ролью Serializable

let serializer = service.Serializable();

let data = serializer.toJSON();

// Передача в функции

func saveToFile(s: Serializable, filename: String) {

`    `File.write(filename, s.toJSON());

}

saveToFile(service.Serializable(), "users.json");  // явно берём нужную роль

## **13. Резюме**

|Механизм|Синтаксис|Назначение|
| :- | :- | :- |
|Пустой метод|func name();|Объявление контракта|
|Наследование|class A : B|Одиночное наследование, иерархия "is-a"|
|Роль|class A : B as C as D|Обязательство выполнить контракты C и D|
|Блок роли|C { ... }|Реализация методов контракта C|
|Представление|obj.C()|Доступ к объекту в роли C|
|Проверка роли|obj is C|Проверка, реализует ли объект роль C|
|Явное приведение|obj as C|Получение представления с проверкой|
|Копирование|obj.clone()|Глубокое копирование объекта|
